# Consuming MCP Bundles in an External Project

This guide describes how to use the UMD bundles (`mcp-core.js`, `mcp-server.js`, `mcp-babylon.js`, `mcp-cesium.js`, etc.) or the TypeScript packages from `mcp-for-babylon` in a **separate** project.

---

## Available Artifacts

| Artifact | Format | Global (UMD) | Notes |
|---|---|---|---|
| `mcp-core.js` | UMD bundle | `window.McpCore` | Interfaces, base classes, grammar, transport |
| `mcp-server.js` | UMD bundle | `window.McpServer` | McpServer, McpServerBuilder |
| `mcp-babylon.js` | UMD bundle | `window.McpBabylon` | Babylon.js adapters (externalizes `@babylonjs/core`) |
| `mcp-cesium.js` | UMD bundle | `window.McpCesium` | Cesium adapters (externalizes `cesium`) |
| `mcp-filters.js` | UMD bundle | `window.McpFilters` | Filter behaviors |
| `@dev/core` | npm package | — | TypeScript source + compiled `dist/` |
| `@dev/babylon` | npm package | — | TypeScript source + compiled `dist/` |
| `@dev/cesium` | npm package | — | TypeScript source + compiled `dist/` |

Bundles are generated by running `npm run build:all` from the monorepo root.
They land in `packages/host/www/bundle/` after the deploy step.

---

## Option 1 — `<script>` Tags (UMD Bundles)

The simplest approach. Copy the `.js` files to your project's static assets folder and load them in order.

```html
<!-- Engine (loaded from CDN or locally) -->
<script src="https://cdn.babylonjs.com/babylon.js"></script>

<!-- MCP bundles — order matters -->
<script src="/lib/mcp-core.js"></script>
<script src="/lib/mcp-server.js"></script>
<script src="/lib/mcp-babylon.js"></script>

<script>
    // Globals are available
    const builder = new McpServer.McpServerBuilder();
</script>
```

**Pros** : Zero tooling, works in any HTML page.
**Cons** : Manual copy, no versioning, no tree-shaking.

**Tip** : Extend the existing `scripts/deploy-bundles.mjs` to copy bundles to your external project automatically:

```js
const EXTERNAL_DEST = resolve(__root, "../../my-other-project/public/lib");
SOURCES.push(/* ... */);
```

## Option 2 — `file:` Protocol (Local npm Link)

Reference the packages directly from disk in your consuming project's `package.json`:

```jsonc
// my-other-project/package.json
{
    "dependencies": {
        "@dev/core": "file:../mcp-for-babylon/packages/dev/core",
        "@dev/babylon": "file:../mcp-for-babylon/packages/dev/babylon"
    }
}
```

Then import normally:

```ts
import { McpServerBuilder } from "@dev/core";
import { BabylonCameraAdapter } from "@dev/babylon";
```

**Pros** : Live changes, TypeScript types included, tree-shakable.
**Cons** : Relative path is fragile across machines; both repos must be side by side.

> **Important** : Run `npm run build:dev` in the monorepo first so the `dist/` folders exist.

---

## Option 3 — `npm pack` (Local Tarball)

Generate a `.tgz` archive and install it like a regular dependency.

```bash
# In the monorepo — build then pack
npm run build:dev
cd packages/dev/core
npm pack
# → dev-core-0.1.0.tgz
```

```bash
# In the consuming project
npm install ../mcp-for-babylon/packages/dev/core/dev-core-0.1.0.tgz
```

**Pros** : No symlinks, works with `"private": true`, version is frozen.
**Cons** : Must re-pack after every change.

---

## Option 4 — `npm link` (Global Symlink)

```bash
# In the monorepo
cd packages/dev/core && npm link
cd ../babylon && npm link

# In the consuming project
npm link @dev/core @dev/babylon
```

**Pros** : Changes reflected instantly, no rebuild needed on the consumer side.
**Cons** : Symlinks can confuse bundlers; singleton libraries (e.g. `@babylonjs/core`) may be duplicated.

## Option 5 — GitHub Packages (Private npm Registry)

Publish to GitHub's built-in npm registry. The repo is already on GitHub (`pandaGaume/mcp-browser-world`).

### 5.1 — Prepare Packages for Publishing

Remove `"private": true` and add a `publishConfig` in each package:

```jsonc
// packages/dev/core/package.json
{
    "name": "@pandagaume/mcp-core",
    "version": "0.1.0",
    "publishConfig": {
        "registry": "https://npm.pkg.github.com"
    },
    "files": ["dist", "src"]
}
```

### 5.2 — Authenticate Once

Create a GitHub **Personal Access Token** (classic) with the `write:packages` scope, then:

```bash
npm login --registry=https://npm.pkg.github.com
# Username: pandaGaume
# Password: <your token>
```

### 5.3 — Publish

```bash
npm run build:dev
npm publish --workspace=packages/dev/core
npm publish --workspace=packages/dev/babylon
npm publish --workspace=packages/dev/cesium
```

### 5.4 — Consume in Another Project

Create a `.npmrc` at the root of the consuming project:

```ini
# .npmrc
@pandagaume:registry=https://npm.pkg.github.com
//npm.pkg.github.com/:_authToken=${GITHUB_TOKEN}
```

Then install normally:

```bash
npm install @pandagaume/mcp-core @pandagaume/mcp-babylon
```

```ts
import { McpServerBuilder } from "@pandagaume/mcp-core";
```

**Pros** : Standard npm workflow, semantic versioning, CI/CD friendly, private by default.
**Cons** : Initial setup (token, `.npmrc`), must publish after each release.

---

## Option 6 — Git Submodule

Add the monorepo as a submodule in the consuming project:

```bash
git submodule add https://github.com/pandaGaume/mcp-browser-world.git vendor/mcp
```

Then either:
- Reference the TypeScript sources via `tsconfig.json` paths
- Or copy the pre-built bundles from `vendor/mcp/packages/host/www/bundle/`

**Pros** : Version pinned to a specific commit.
**Cons** : Submodule workflow is heavy (init, update, sync).

## Quick Decision Matrix

| Scenario | Recommended Option |
|---|---|
| Quick HTML prototype | **Option 1** — `<script>` tags |
| TypeScript project, local dev | **Option 2** — `file:` protocol |
| Share a snapshot with a colleague | **Option 3** — `npm pack` |
| Active parallel development | **Option 4** — `npm link` |
| Team / CI / production | **Option 5** — GitHub Packages |
| Vendoring into a monorepo | **Option 6** — Git submodule |

---

## Bundle Load Order

When using UMD bundles via `<script>` tags, respect this dependency order:

```
1. Engine          (babylon.js or Cesium.js)
2. mcp-core.js     (no dependency)
3. mcp-server.js   (depends on mcp-core)
4. mcp-filters.js  (depends on mcp-core)        — optional
5. mcp-babylon.js  (depends on mcp-core + engine) — OR mcp-cesium.js
```

---

## Notes

- All packages currently have `"private": true`. This blocks `npm publish` but **not** `npm pack`, `file:`, or `npm link`.
- The `"files"` field in each `package.json` already includes `["dist", "src"]`, so both compiled JS and TypeScript sources are shipped.
- UMD bundles externalize engine libraries (`@babylonjs/core`, `cesium`) — the consumer must provide them separately.
